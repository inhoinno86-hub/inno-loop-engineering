Loop Engine — 팀 공유 문서

작성일: 2026-07-23  |  버전: Phase 1~4 통합 완료 (Shared Core + 2 Frontends)

1. 배경 및 목적

개발 요청을 받으면 기획 → 계획 → 구현 → 검토 → 재계획 사이클이 반복된다. 이 반복을 사람이 수동으로 관리하면 컨텍스트 전환 비용이 크고, 중간 단계에서 누락이 생긴다.

Loop Engine은 이 사이클 전체를 Claude Code가 오케스트레이션하도록 자동화한 시스템이다. 사용자는 project_name과 user_intent(의도 문서)만 전달하면, 나머지 init → plan → run → review 4단계가 사람 개입 없이 반복 실행된다.

2. 전체 아키텍처

Shared Core + 2 Frontends 구조: loop_engine pip 패키지가 Shared Core(단일 진실 출처)이며, Claude Code(사내 환경)와 Codex CLI(개인 PC) 두 Frontend가 동일한 CLI를 호출한다. 핵심 정책 스펙은 loop_engine/docs/spec/에 위치한다.

4. Human-in-the-Loop (HLT) 조건

기본 전제는 사람 개입 없이 루프를 완주하는 것이다. 단, 아래 6가지 조건이 발생하면 시스템이 멈추고 사용자에게 판단을 요청한다.

ID조건명트리거
HLT-1Socrates 신뢰도 부족required_field_coverage < 0.7 또는 self_consistency_score < 50%
HLT-2Mediator 수렴 실패Mediator 개입 후에도 의사결정 불능
HLT-3Plan 방향 충돌PLAN이 user_intent와 명시적으로 상충
HLT-4회귀 횟수 초과regression_count ≥ 5
HLT-5Critical 이슈보안 취약점, 데이터 손상, 되돌릴 수 없는 작업
HLT-6Correction 초과correction_attempts ≥ 3

HLT 발생 시: 현재 단계 상태 보존 → halt → 사용자 입력 대기 → 재개 가능.

5. Agent 구성

Agent역할실행 시점
Socrates 가설 생성 / 계획 정제 / 수정. 3가지 모드 자동 전환 init (N=3 병렬), plan (최초·회귀), correction
Plan Validator user_intent 직접 파싱 → coverage_score 계산 plan 확정 직후 (run 진입 gate)
Mediator 충돌 항목 조율, 최종 결정 중재 self_consistency 50~80% 구간, correction attempt 2
Evaluator PLAN 기준 항목별 pass/fail 판정 project-review

Socrates 신뢰도 게이트 (init 단계)

N=3 독립 인스턴스 병렬 실행 → 결과 집계

Gate 1: required_field_coverage < 0.7     → HLT-1
Gate 2: uncertainty_marker > 5/문서        → HLT-1
Gate 3: self_consistency_score
  < 50%:    → HLT-1
  50~80%:   → Mediator 검토
  > 80%:    → 자율 결정 허용

6. project-run 내부 흐름

1. 환경 준비
   - 최신 PLAN_v{N}.md 확인
   - git worktree 격리 (브랜치: loop-engine/{name}/run-v{N})

2. make-prompts 실행
   - /make-prompts 스킬 호출 → traceability_prompts/ 생성
   - 완료 즉시 사용자 개입 없이 Step 3 자동 진행

3. exec-prompts 실행
   - /exec-prompts 스킬 호출 → sub-agent 단계별 실행
   - 각 agent spawn 시 registry add + heartbeat touch
   - 60초 주기 orphan 감시

4. 완료 처리
   - registry update --status completed
   - loop-engine state write --stage project-review
   - project-review 자동 진입

7. Python 유틸 패키지 (loop_engine CLI)

# 상태 관리
loop-engine state init   --project <name> --intent-file <path>
loop-engine state read   --project <name>       # stdout: 현재 stage
loop-engine state write  --project <name> --stage <stage>

# Agent 레지스트리
loop-engine registry add    --project <name> --agent-id <id> --parent <id> --depth <n> --task-scope <scope>
loop-engine registry update --project <name> --agent-id <id> --status <status>
loop-engine registry list   --project <name>
loop-engine registry orphan-check --project <name> --timeout 600

# Heartbeat
loop-engine heartbeat touch  --project <name> --agent-id <id>
loop-engine heartbeat clean  --project <name>

# Coverage (LLM 기반)
loop-engine coverage score --project <name> --plan-file <path> --intent-file <path>
# stdout: float 0.0~1.0

# 세션 아카이브
loop-engine archive session --project <name> --timestamp <YYYYMMDD_HHMMSS>

11. Approval 5-카테고리 (BLOCKED 조건)

자동 진행 중 아래 중 하나 해당 시 즉시 BLOCKED — 사용자 명시 승인 없이 재개 불가:

카테고리조건
external_irreversible외부/비가역 효과
security_privacy_secrets보안·개인정보·시크릿
budget_limit_breach예산·모델·실행 한도 초과
intent_or_core_architecture_change의도·핵심 설계 변경
repeated_evaluation_failure동일 실패 3회 연속
  • uncertain_risk → fail-closed → BLOCKED 유지
  • Timeout ≠ 승인. Silence = BLOCKED 유지
