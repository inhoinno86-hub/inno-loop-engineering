현재 변경사항 비교 분석

작성일: 2026-07-24 · 대상 저장소: inno-loop-engineering

결론: 현재 변경은 “inno-loop를 호출한 뒤 사람이 매 단계마다 승인하지 않아도 전체 생명주기를 계속 실행하게 하는 정책”을 추가합니다. 동시에 초기화·계획 단계에서 3개 통합 도구의 실행 증거를 반드시 남기고, 리뷰 실패 시 자동 재계획을 최대 3회까지 반복하도록 바꿉니다.

1. 비교 기준과 범위

날짜 확인 결과: 2026-07-23(어제)에는 커밋이 없습니다. 따라서 요청한 “어제 작업이 정답”이라는 가정은 가장 최근 커밋 e874a97 (2026-07-22, feat: replace project loop with inno loop engineering)을 정답 기준선으로 삼아 적용했습니다.

변경량—+300줄 / -33줄
스테이징—없음. 모두 미스테이지 변경
핵심 코드기본 4단계 상태 전이통합 증거, 전체 실행 권한, 재계획 반복 제한 추가

2. 한눈에 보는 동작 변화

단계기준선 동작현재 동작
초기화 → 계획초기화 산출물 증거만 있으면 다음 단계로 이동Ouroboros interview, Superpowers brainstorming, writing-plans의 성공 증거 3개가 있어야 이동
계획 → 실행계획 증거가 있으면 실행으로 이동계획 단계의 3개 통합 증거와 전체 생명주기 권한이 모두 필요
리뷰 실패REPLAN이 별도 결과값으로 기록REPLAN은 종료하지 않는 전이. 곧바로 계획 단계로 돌아가 새 반복을 시작
반복 한도제한 없음기본 3회. 그 뒤 재계획이 또 필요하면 BLOCKED

쉽게 말하면: 기존은 “다음 단계로 갈 수 있는지”를 주로 관리했습니다. 현재 변경은 “정해진 도구를 실제로 썼는지, 전체 자동 진행을 허용했는지, 실패 시 언제 멈출지”까지 상태 파일에 남겨 관리합니다.

4. 정답 기준선 관점의 평가

일치정책·코드·테스트의 큰 방향은 일치합니다.

문서에서 말한 “통합 증거 없이는 진행 불가”, “전체 생명주기 권한 필요”, “REPLAN은 비종료 전이”, “반복 횟수 제한”이 상태 엔진과 CLI, 테스트에 모두 반영돼 있습니다.

호환성기존 REPLAN 상태를 읽을 수 있게 처리했습니다.

이전 상태 파일에 outcome: REPLAN이 남아 있으면 읽을 때 현재 형식으로 바꾸고, 재계획 이력과 계획 반복 번호를 보완합니다. 기존 실행 기록을 바로 버리지 않는 안전한 방식입니다.

의도된 엄격화도구가 없으면 일반 Codex 방식으로 대체하지 않고 중단합니다.

기존보다 작업이 멈출 가능성은 커집니다. 그러나 이번 변경의 명시된 목표가 “필수 통합을 강제하고 증거를 남기는 것”이므로, 정답 기준선의 방향을 따른 변화로 평가했습니다.

5. 확인된 차이·리스크

중간문서의 “순서대로 실행” 규칙을 코드가 검사하지 않습니다.

문서는 interview → brainstorming → writing-plans 순서를 요구하지만, required_integrations_used()는 세 이름이 존재하는지만 집합(set)으로 확인합니다. 따라서 CLI에서는 writing-plans를 먼저 기록해도 통과합니다. 순서가 감사·재현성에 중요하다면 기록 순서 또는 타임스탬프를 검증하는 로직과 테스트가 필요합니다.

근거: scripts/loop_engine.py:171-178, skills/inno-loop/SKILL.md

중간재계획 때 “새 버전 산출물”을 강제하지 않습니다.

문서는 재계획마다 새 계획 산출물과 새 통합 증거를 요구합니다. 하지만 CLI는 프로젝트 루트 안의 어떤 파일이든 증거로 받고, 이전 반복과 같은 파일 경로·같은 해시도 다시 등록할 수 있습니다. 즉 문서상 규칙은 안내 수준이며 강제되지는 않습니다.

근거: scripts/loopctl.py:35-50, scripts/loop_engine.py:171-178

낮음상태 템플릿이 런타임 상태의 모든 새 필드를 보여 주지는 않습니다.

state.json 템플릿에는 plan_iteration과 replan_count가 없습니다. 실제 initialize()는 두 필드를 생성하고 load()도 기본값을 보완하므로 실행 오류는 아닙니다. 다만 템플릿을 상태 계약의 예시로 쓰는 초보자에게는 혼동을 줄 수 있습니다.

근거: assets/templates/state.json, scripts/loop_engine.py:96-103

7. 최종 판단

현재 변경은 최근 커밋의 inno-loop 구조를 확장하는 방향으로 일관적이며, 테스트도 통과했습니다. 특히 자동 재계획을 “종료 결과”에서 “다음 계획 반복”으로 바꾼 점과, 통합 사용 증거를 상태에 남긴 점이 핵심 개선입니다.

다만 이 정책을 엄격한 운영 계약으로 사용할 계획이라면, 커밋 전에는 위의 두 중간 리스크(도구 실행 순서, 재계획 시 새 산출물 강제)를 보완할지 결정하는 것이 좋습니다.
